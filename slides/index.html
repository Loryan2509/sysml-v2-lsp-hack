The Problem Space

Systems Models Are Too Complex
for Humans Alone

SysML v2 models can describe entire aerospace, automotive, or medical systems — thousands of interconnected parts, actions, states, and requirements. Spotting logical flaws, missing failure modes, or conflicting constraints requires deep, multi-domain reasoning that human reviewers routinely miss.

💥
Human Review Bottleneck
Highly trained systems engineers spend weeks reviewing models that LLMs can scan in seconds — and still miss subtle emergent issues.
🔗
No Rich Open Tooling
Most SysML tools are proprietary, heavyweight, or lack any AI integration. Open-source ecosystems for SysML v2 were nearly non-existent.
🤖
LLMs Lack Model Context
Raw LLMs have no structured access to SysML model data. They can't navigate definitions, validate types, or compute complexity metrics.

In Action

LLM Finds What Humans Miss

🗂️ Example SysML v2 Model
  drone-system.sysml
package DroneSystem {
  part def Drone {
    part motor1 : Motor;
    part motor2 : Motor;
    part battery : Battery;
    part gps : GPS;
    // ← No failure mode defined
    // ← No redundancy for motors
  }
  action def Fly {
    // ← No return-to-home action
  }
}
🤖 AI Assessment (via review-sysml)
✅ Completeness
0.55
🛡️ Failure Modes & Resilience
0.80
🎯 Correctness & Intent
0.20
🌊 Emergent Properties
0.65
⚔️ Conflicting Requirements
0.10
⚠️ LLM Finding — HIGH Confidence
Missing failure modes for motor loss. A 2-motor drone with no failsafe has a critical single point of failure. Missing: geofencing, return-to-home action, and motor-loss recovery state machine.
⚠️ LLM Finding — MEDIUM Confidence
Battery capacity not modelled — no constraint links power draw to flight time. This is an emergent system property the model cannot currently verify.
